Proposal: CID-Native, Capability-Aware, P2P Execution Profile for MCP

Status: Draft / Discussion
Target audience: MCP maintainers, Agentic AI Foundation (AAIF) contributors, implementers
Scope: Optional extension profile (non-breaking)

1. Executive Summary

The Model Context Protocol (MCP) standardizes structured tool invocation between models and external systems using JSON-RPC over extensible transports. This proposal introduces an optional execution profile that enables MCP deployments to operate safely and audibly in federated, multi-agent, and parallel execution environments.

The proposal does not modify MCP message semantics. Instead, it defines:

  • a peer-to-peer transport binding,

  • a content-addressed execution envelope,

  • a capability-based delegation model,

  • and a formal policy evaluation layer for time-bounded permissions and obligations.

The goal is to support delegation, provenance, auditability, and concurrency control in distributed MCP ecosystems while preserving full backward compatibility.

2. Motivation

MCP is highly effective for:

  • IDE integrations

  • Single-authority tool servers

  • Client↔server interaction patterns

However, emerging deployments increasingly involve:

  • multiple autonomous agents,

  • delegation across organizational or trust boundaries,

  • parallel tool execution,

  • long-running workflows,

  • and partial connectivity.

In these environments, implementers repeatedly encounter the same unanswered questions:

  • Who was authorized to perform this action?

  • Under what conditions and time constraints?

  • Which exact data was consumed and produced?

  • Can the system replay, audit, or attribute responsibility for outcomes?

These questions are outside MCP’s current scope but are critical for safe large-scale deployment. This proposal addresses them through an opt-in execution profile.

3. Design Goals and Constraints

3.1 Goals

  • Preserve MCP compatibility

  • Enable federated, multi-agent execution

  • Support explicit delegation of authority

  • Provide immutable provenance and audit trails

  • Allow safe parallelism and rollback

  • Enable policy-driven execution decisions

3.2 Non-Goals

  • Replacing MCP transports

  • Introducing mandatory global consensus

  • Enforcing a single policy language

  • Mandating blockchain or ledger infrastructure
